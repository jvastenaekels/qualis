## HIGH severity (release-blocker for docs)

### H-1 — Study lifecycle: translations/metadata are NOT editable once Active
`docs/guides/admin-management.md:51,66,71,72,73` · `docs/reference/admin-dashboard.md:81` · `docs/guides/conducting-studies.md:66`
- **Doc says:** Once Active/Paused/Closed, "translations and metadata remain editable"; Paused is "Use to fix a typo in a statement without taking the study down."
- **Code does:** For any non-draft state the guard allows only `{start_date, end_date}` (empty set when archived); every other field raises *"Cannot update study in {state} state. Switch it back to draft first."*
- **Evidence:** `backend/app/services/study_service.py:206-218`; route `backend/app/routers/admin/studies.py:121` (docstring "draft only").
- **Fix:** Replace all "translations and metadata editable" wording with: *"Once Active, the entire configuration (grid, statements, translations, metadata) is read-only; only the collection-window dates (start/end) are editable. Switch back to Draft to edit anything else."* Delete the Paused "fix a typo" guidance.

### H-2 — S3 audio: enablement needs `S3_ENDPOINT_URL` for **all** providers; `S3_REGION` does not gate
`docs/guides/s3-setup.md:68-73,117,151`
- **Doc says:** Four vars (`S3_BUCKET_NAME`, `S3_REGION`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`) enable audio; `S3_ENDPOINT_URL` is "only for non-AWS."
- **Code does (verified):** `is_s3_configured` = `S3_ENDPOINT_URL AND S3_BUCKET_NAME AND S3_ACCESS_KEY_ID AND S3_SECRET_ACCESS_KEY`; `S3_REGION` is absent (defaults `us-east-1`). Without `S3_ENDPOINT_URL`, audio silently stays disabled on AWS too (router 503, `StorageService.__init__` raises).
- **Evidence:** `backend/app/core/config.py:177-182`; `backend/app/services/storage_service.py:97-107`; banner `backend/app/utils/storage_mode.py:14-15`.
- **Fix:** Make `S3_ENDPOINT_URL` mandatory for every provider (AWS = regional endpoint URL); note `S3_REGION` is optional (default `us-east-1`) and not part of the gate. Correct the "four vars" enumeration to the four *gating* vars.

### H-3 — `SMTP_FROM` env var does not exist

### H-5 — Error envelope `code` is a string identifier, not the HTTP status integer
`docs/reference/api.md:356,343`
- **Doc says:** `code | integer | HTTP status code`; example `"code": 422`.

### H-6 — JWT family model is wrong (access token has no `purpose`/`role`; `sub` is email; invitation uses `type`; decoder name is wrong)
`docs/contributing/backend-guidelines.md:63-67`
- **Doc says:** All three families isolated by a `purpose` claim; access carries `purpose=access` + `role`; invitation `purpose=invitation`; a single `decode_token(token, expected_purpose=...)` enforces it; email purposes are hyphenated; email tokens each carry a single-use `jti`.
- **Code does (verified):** `create_access_token` encodes exactly `{exp, iat, sub}` — **no** purpose, **no** role; `sub` = the user's *email*, not id (auth.py issues `subject=user.email`). Invitation uses a `type` claim. There is **no** `decode_token`; the three decoders are `decode_access_token`/`decode_invitation_token`/`decode_email_token`. Email purposes use underscores: `email_verify`, `password_reset`, `twofa_disable`, `email_change_confirm`, `email_change_cancel`. Every email token carries a `jti`, but only `twofa_disable` is denylisted; `is_jti_consumed()` has zero callers; password_reset relies on the `pwa` claim.
- **Evidence:** `security.py:70-74, 94-103, 110-126, 142-143, 14-20`; `auth.py:233-234, 700-714, 1018`.
- **Fix:** Rewrite §3.1 per C20/C21/C23/C24/C25/C26/C27/C28: access = no discriminator, `sub`=email, role server-side; invitation = `type` claim, not single-use; email = underscore purposes, per-purpose single-use (only `twofa_disable` denylisted); name the three real decoders. **This is the single most internally-wrong section in the docs.**

### H-7 — `init_db.py` is non-destructive without `--reset`; migrations do not run on app startup
`docs/contributing/agent-instructions.md:20,49-50`

- **Evidence:** `backend/init_db.py:122,129-132`; `backend/app/main.py` lifespan; `backend/scripts/migrate.py`.
- **Fix:** "Run `make db-reset` (→ `init_db.py --reset`) to drop/recreate; plain `init_db.py` only migrates+seeds. Migrations run on deploy via the Procfile `postdeploy` phase, not at app startup."
